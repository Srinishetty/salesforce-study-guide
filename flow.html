Types of Flow — When to Use Each

Salesforce offers different flow types for different automation needs. Choose based on whether you need UI interaction or background processing.

Record-Triggered Flow — Timing Options

WhenDescriptionLike in Apex...
Before SaveModify THIS record (no DML cost)before trigger

Before Save is the most efficient — changes happen on the same record with zero additional DML statements.

Flow vs Apex Comparison

// Apex equivalent of a Record-Triggered Flow:
// "When Account is created, set Rating based on Revenue"

// FLOW (declarative):
// Trigger: Account | After Create
// Decision: Revenue > 1M? 
//   Yes → Assignment: Rating = 'Hot'
//   No → Assignment: Rating = 'Warm'

// APEX (code):
trigger AccountTrigger on Account (before insert) {
    for (Account acc : Trigger.new) {
        if (acc.AnnualRevenue > 1000000) {
            acc.Rating = 'Hot';
        } else {
            acc.Rating = 'Warm';
        }
    }
}

Record-Triggered Flow — Common Patterns

Record-Triggered Flows replace Process Builder and Workflow Rules. They fire automatically when records are created, updated, or deleted.

Pattern 1: Field Update (Before Save)

// Scenario: Auto-set Priority based on Amount
// Trigger: Opportunity | Before Save | Create & Update
// Entry Conditions: Amount is changed

// Decision: "Check Amount Range"
//   Outcome 1: Amount > 100000
//     → Assignment: $Record.Priority__c = 'High'
//   Outcome 2: Amount > 50000
//     → Assignment: $Record.Priority__c = 'Medium'
//   Default:
//     → Assignment: $Record.Priority__c = 'Low'

// ✅ No DML needed! Before-save modifies $Record directly
// ✅ Most efficient — zero additional queries/DML

Pattern 2: Create Related Records (After Save)

// Scenario: Create a Task when Opp Stage = 'Proposal'
// Trigger: Opportunity | After Save | Update only
// Entry: Stage is changed AND Stage = 'Proposal'

// Create Records:
//   Object: Task
//   Fields:
//     WhatId = {!$Record.Id}
//     Subject = 'Send Proposal to ' + {!$Record.Name}
//     OwnerId = {!$Record.OwnerId}
//     ActivityDate = {!$Flow.CurrentDate} + 3
//     Priority = 'High'

// ✅ Runs after commit — record has ID
// ✅ Can create/update OTHER objects

Pattern 3: Roll-Up Summary (After Save)

// Scenario: Count open Cases on Account
// Trigger: Case | After Save | Create, Update, Delete

// Get Records: Query all open Cases for this Account
//   Object: Case
//   Conditions: AccountId = {!$Record.AccountId}
//               AND IsClosed = false
//   Store: {!openCases} (collection)

// Assignment:
//   {!caseCount} = {!openCases}.size()  (formula)

// Update Records:
//   Record: Account where Id = {!$Record.AccountId}
//   Field: Open_Cases__c = {!caseCount}

// ⚠️ Watch for recursion! Enable "Run once per record"

Pattern 4: Send Notification (After Save, Async)

// Scenario: Email manager when deal > $500K closes
// Trigger: Opportunity | After Save (Async) | Update
// Entry: StageName is changed to 'Closed Won'
//        AND Amount > 500000

// Get Records: Get Owner's Manager
//   Object: User
//   Conditions: Id = {!$Record.OwnerId}
//   Store: {!ownerUser}

// Action: Send Email
//   To: {!ownerUser.ManagerId}
//   Subject: 'Big Deal Closed: ' + {!$Record.Name}
//   Body: Template with merge fields

// ✅ Async = won't slow down the save
// ✅ Good for: emails, callouts, heavy processing

Flow Best Practices & Performance

Flows can hit governor limits just like Apex. Bulkification matters here too.

Performance Rules

  • No DML/SOQL inside loops! — Same rule as Apex. Put Get/Create/Update OUTSIDE the loop element.
  • Use Before Save when modifying the same record — costs zero DML.
  • One flow per object per timing — don't create 5 separate after-save flows on Account.
  • Use Entry Conditions — only run when relevant fields change (isChanged formula).
  • Bulkify manually — collect records in a loop into a collection variable, then do ONE Create/Update after the loop.

Bulkification Example

// ❌ BAD: DML inside loop
// Loop over Contacts:
//   → Create Record (Task) ← INSIDE LOOP!
// Result: 200 records = 200 DML = LIMIT HIT

// ✅ GOOD: Collect + batch DML
// Create variable: {!tasksToCreate} (Record Collection)
// Loop over Contacts:
//   → Assignment: Add new Task to {!tasksToCreate}
// AFTER loop:
//   → Create Records: {!tasksToCreate}  ← ONE DML!

// Same pattern for Updates:
// Loop → Assignment (add to collection) → Update AFTER loop

Debugging Flows

  • Debug Mode — Run in Flow Builder with test inputs, see each step's result
  • Flow Debug Logs — Setup → Debug Logs → shows FLOW_START_INTERVIEW
  • Fault Email — Auto-sent to admin with full error details
  • $Flow.FaultMessage — Access error text in fault paths
  • Pause Element — Pause and resume (scheduled actions, wait for conditions)

Invoke Apex from Flow (Action)

// Create an Invocable Apex method:
public class FlowActions {
    
    @InvocableMethod(
        label='Calculate Discount'
        description='Returns discount % based on rules'
    )
    public static List<Decimal> calculateDiscount(
        List<DiscountRequest> requests
    ) {
        List<Decimal> results = new List<Decimal>();
        for (DiscountRequest req : requests) {
            if (req.customerTier == 'Platinum') {
                results.add(20.0);
            } else if (req.totalAmount > 10000) {
                results.add(10.0);
            } else {
                results.add(5.0);
            }
        }
        return results;
    }
    
    public class DiscountRequest {
        @InvocableVariable(required=true)
        public String customerTier;
        @InvocableVariable
        public Decimal totalAmount;
    }
}

// In Flow: Action element → "Calculate Discount"
// Inputs auto-map from flow variables!

Flow + LWC Integration

// Embed a Flow in LWC:
<template>
    <lightning-flow 
        flow-api-name="Create_Case_Wizard"
        flow-input-variables={inputVars}
        onstatuschange={handleFlowStatus}>
    </lightning-flow>
</template>

// JS:
get inputVars() {
    return [
        { name: 'recordId', type: 'String', value: this.recordId },
        { name: 'accountName', type: 'String', value: this.accName }
    ];
}

handleFlowStatus(event) {
    if (event.detail.status === 'FINISHED') {
        // Flow completed — get output variables
        const outputVars = event.detail.outputVariables;
        const caseId = outputVars.find(v => v.name === 'newCaseId');
        this.navigateToRecord(caseId.value);
    }
}

// Set target: lightning__FlowScreen in meta.xml
// Use @api properties for flow input/output
// Dispatch FlowAttributeChangeEvent for outputs
