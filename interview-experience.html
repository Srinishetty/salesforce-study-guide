💻 Technical Scenarios & Apex

Scenario: Prevent a user from deleting a Contact if it is the last Contact associated with an Account.

Approach: Use a `before delete` trigger on Contact. Collect Account IDs, query those Accounts with a subquery for their Contacts, and if the list size is 1 or less, use `.addError()` on the trigger record.

trigger PreventLastContactDeletion on Contact (before delete) {
    Set<Id> accountIds = new Set<Id>();
    for (Contact con : Trigger.old) {
        if (con.AccountId != null) { accountIds.add(con.AccountId); }
    }

    if (!accountIds.isEmpty()) {
        Map<Id, Account> accountMap = new Map<Id, Account>([
            SELECT Id, (SELECT Id FROM Contacts) 
            FROM Account WHERE Id IN :accountIds
        ]);

        for (Contact con : Trigger.old) {
            if (con.AccountId != null) {
                Account relatedAcc = accountMap.get(con.AccountId);
                if (relatedAcc != null && relatedAcc.Contacts.size() <= 1) {
                    con.addError('Cannot delete the last associated contact.');
                }
            }
        }
    }
}

Scenario: When a Lead is converted, automatically create a Task assigned to the owner of the Account.

Answer: "I would use a Record-Triggered Flow on the Lead object (triggered on Update). The condition would be `IsConverted Equals True`. I would then use a Get Records element to find the Account where `Id = $Record.ConvertedAccountId`, followed by a Create Records element to create the Task, setting the `OwnerId` to the Account Owner's ID."

⚡ LWC & UI Architecture

Scenario: Build a custom search button on a page layout that extracts records from multiple objects (Account, Contact) based on an email.

Answer: "I would build a custom LWC exposed as a Quick Action. The LWC takes an email input and calls an `@AuraEnabled` Apex method imperatively. Inside Apex, I would use SOSL (`FIND {email} IN Email FIELDS RETURNING Account(Id, Name), Contact(Id, Name)`), and return the list of lists back to the LWC to render."

How do you pass data from a parent component down to a grandchild component in LWC?

Answer: "There are two ways. The first is 'Prop Drilling', where the Parent passes data to the Child using an `@api` property, and the Child immediately passes it to the Grandchild. The second, better approach for deeply nested components is Lightning Message Service (LMS), where the Parent publishes a message and the Grandchild subscribes directly."
