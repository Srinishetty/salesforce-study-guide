Tech Lead

You have an Account associated with multiple Contacts. A user should not be able to delete a Contact if it is the "last associated" one. Write the code.

🎤 How to answer naturally
You (Acting as a Senior Developer clarifying requirements): "In a real project, I would immediately ask the business to clarify 'last associated', because it can mean two very different things technically." "If they mean the Final Remaining Contact—meaning the Account must never be left empty—I would use an Aggregate query in a before delete trigger. I'd count the total current contacts, subtract the ones being deleted, and if that number drops below 1, I use .addError(). This guarantees even a mass-delete won't leave the Account with zero contacts." "But if they mean the Chronologically Most Recently Linked Contact, we have a standard platform limitation. A Contact can be created on Monday, but linked to the Account on Friday. Because of that, the standard CreatedDate is useless." "To properly solve this, I would create a custom DateTime field called Account_Linked_Date__c. I'd stamp it via Flow when the AccountId changes. Then, in my before delete trigger, I would query the contacts ordered by that custom date descending. If the user tries to delete that specific ID, I block it."
Code: Scenario 1 (Preventing Account from being empty - Mass Delete Safe)
trigger PreventEmptyAccount on Contact (before delete) {
    // 1. Map Account Id to count of contacts being deleted
    Map<Id, Integer> deletedCount = new Map<Id, Integer>();
    for(Contact c : Trigger.old) {
        if(c.AccountId != null) {
            Integer count = deletedCount.containsKey(c.AccountId) ? deletedCount.get(c.AccountId) : 0;
            deletedCount.put(c.AccountId, count + 1);
        }
    }
    
    if(!deletedCount.isEmpty()) {
        // 2. Aggregate query for total current contacts
        Map<Id, AggregateResult> totalMap = new Map<Id, AggregateResult>([
            SELECT AccountId, COUNT(Id) total FROM Contact 
            WHERE AccountId IN :deletedCount.keySet() GROUP BY AccountId
        ]);
        
        Set<Id> invalidAccounts = new Set<Id>();
        for(Id accId : deletedCount.keySet()) {
            Integer total = totalMap.get(accId) != null ? (Integer)totalMap.get(accId).get('total') : 0;
            if((total - deletedCount.get(accId)) < 1) {
                invalidAccounts.add(accId); // Drops below 1
            }
        }
        
        for(Contact c : Trigger.old) {
            if(invalidAccounts.contains(c.AccountId)) {
                c.addError('Cannot delete: Account must retain at least one Contact.');
            }
        }
    }
}
Code: Scenario 2 (Preventing deletion of the Most Recently Linked)
trigger PreventNewestLinkDeletion on Contact (before delete) {
    Set<Id> accIds = new Set<Id>();
    for(Contact c : Trigger.old) {
        if(c.AccountId != null) accIds.add(c.AccountId);
    }
    
    if(!accIds.isEmpty()) {
        Set<Id> protectedContactIds = new Set<Id>();
        // Query the most recently linked contact using our custom field
        for(Account acc : [SELECT Id, 
            (SELECT Id FROM Contacts ORDER BY Account_Linked_Date__c DESC LIMIT 1) 
            FROM Account WHERE Id IN :accIds]) {
            if(!acc.Contacts.isEmpty()) {
                protectedContactIds.add(acc.Contacts[0].Id);
            }
        }
        
        for(Contact c : Trigger.old) {
            if(protectedContactIds.contains(c.Id)) {
                c.addError('Cannot delete the most recently associated Contact.');
            }
        }
    }
}

Tech Lead

When a Lead is converted, a task should automatically be created for the owner of the Account. Walk me through the Flow solution, and then tell me how you would write it in Apex if required.

🎤 How to answer naturally
You: "The absolute best practice today is a Record-Triggered Flow. I'd set it to fire After-Save on the Lead object, with the condition `IsConverted = True` and set it to only fire when the record is updated to meet the condition. Then I'd drop in a 'Create Records' element for the Task, mapping `WhatId` to the Lead's `ConvertedAccountId` and `OwnerId` to the `Account.OwnerId`." "However, if the org has a strict 'Apex Only' framework, I would write this in an after update trigger on Lead. I'd loop through the updated Leads, check if IsConverted just changed to true, collect the converted Account IDs, query their owners, and insert the Task list."
Solution 1: Record-Triggered Flow (Best Practice)
  • Trigger: A record is Updated. Object: Lead. Optimize for: Actions and Related Records (After-Save).
  • Condition: IsConverted Equals $GlobalConstant.True (Select: Only when a record is updated to meet condition).
  • Create Records Element: Create 1 record of type Task.
  • Field Mappings:
    - WhatId = $Record.ConvertedAccountId
    - OwnerId = $Record.ConvertedAccountId (Assuming standard mapping, or use a Get Records on Account to fetch OwnerId)
    - Subject = "Follow up on Converted Lead"
Solution 2: Apex Trigger (Alternative)
trigger LeadConvertTask on Lead (after update) {
    List<Task> tasksToInsert = new List<Task>();
    Set<Id> accountIds = new Set<Id>();
    
    // Identify freshly converted leads
    for(Lead l : Trigger.new) {
        Lead oldLead = Trigger.oldMap.get(l.Id);
        if(l.IsConverted && !oldLead.IsConverted && l.ConvertedAccountId != null) {
            accountIds.add(l.ConvertedAccountId);
        }
    }
    
    if(!accountIds.isEmpty()) {
        // Query Account owners
        Map<Id, Account> accMap = new Map<Id, Account>([SELECT Id, OwnerId FROM Account WHERE Id IN :accountIds]);
        
        for(Lead l : Trigger.new) {
            if(l.IsConverted && l.ConvertedAccountId != null) {
                tasksToInsert.add(new Task(
                    Subject = 'Follow up with new Account',
                    WhatId = l.ConvertedAccountId,
                    OwnerId = accMap.get(l.ConvertedAccountId).OwnerId,
                    Status = 'Not Started'
                ));
            }
        }
        if(!tasksToInsert.isEmpty()) insert tasksToInsert;
    }
}

public with sharing class GlobalSearchController {
    @AuraEnabled(cacheable=true)

        String searchQuery = 'FIND \'' + emailStr + '\' IN EMAIL FIELDS RETURNING Account(Id, Name), Contact(Id, Name)';
        return search.query(searchQuery);
    }
}

How do you handle component communication? specifically from a Parent down to a Grandchild component?

🎤 How to answer naturally
You: "If I'm just passing data down one level to a child, I use an @api property. But for a grandchild component, 'prop drilling' (passing from parent -> child -> grandchild) gets messy and tightly couples the components." "Instead, I would use Lightning Message Service (LMS). I'd create a Message Channel. The parent publishes the payload, and the grandchild subscribes to it directly, bypassing the middle child entirely."
Code: Parent Component (Publisher)
import { publish, MessageContext } from 'lightning/messageService';
import DATA_CHANNEL from '@salesforce/messageChannel/DataChannel__c';

export default class Parent extends LightningElement {
    @wire(MessageContext) messageContext;

    sendData() {
        const payload = { recordId: '001...', status: 'Active' };
        publish(this.messageContext, DATA_CHANNEL, payload);
    }
}
Code: Grandchild Component (Subscriber)
import { subscribe, MessageContext } from 'lightning/messageService';
import DATA_CHANNEL from '@salesforce/messageChannel/DataChannel__c';

export default class Grandchild extends LightningElement {
    @wire(MessageContext) messageContext;
    subscription = null;

    connectedCallback() {
        this.subscription = subscribe(this.messageContext, DATA_CHANNEL, (message) => {
            console.log('Received ID: ', message.recordId);
        });
    }
}

Architect

How do you restrict specific internal domains in Email-to-Case? Like preventing internal `gmail@tcs.com` emails from creating cases? Explain both Flow and Apex solutions.

🎤 How to answer naturally
You: "Out of the box, standard Email-to-Case ingests everything. The easiest declarative way to stop this is a Before-Save Record-Triggered Flow on the Case object. We check if SuppliedEmail contains '@tcs.com' and auto-close it." "However, if the volume of spam is massive, creating and closing records wastes database space. In that case, I would write a custom Apex Email Service. This runs *before* a record is ever created. I can inspect the sender's address in memory, and if it's internal, I just exit the script without running any DML."
Solution 1: Before-Save Flow (Quick Setup)
  • Trigger: Record is Created. Object: Case. Optimize for: Fast Field Updates (Before-Save).
  • Condition: SuppliedEmail Contains @tcs.com
  • Action (Update Records): Use the flow that triggered the record. Set Status = Closed.
Solution 2: Apex Email Service (Total Control / Saves Database Storage)
global class BlockInternalEmailHandler implements Messaging.InboundEmailHandler {
    global Messaging.InboundEmailResult handleInboundEmail(Messaging.InboundEmail email, Messaging.InboundEnvelope envelope) {
        Messaging.InboundEmailResult result = new Messaging.InboundEmailResult();
        
        // Check the sender domain in memory BEFORE doing any DML
        if(email.fromAddress.endsWithIgnoreCase('@tcs.com')) {
            result.success = true; // Return true so it doesn't bounce back to sender
            return result;         // Exit without creating a Case
        }
        
        // Otherwise, process normally
        Case newCase = new Case(
            Subject = email.subject,
            Description = email.plainTextBody,
            SuppliedEmail = email.fromAddress
        );
        insert newCase;
        
        result.success = true;
        return result;
    }
}

What is the difference between Platform Events and Change Data Capture (CDC)? How are they implemented?

🎤 How to answer naturally
You (Drawing the distinction): "They are both event-driven architectures running on the Salesforce Event Bus, but CDC is declarative data-sync, while Platform Events are custom business logic." "CDC is a standard feature. I literally just click a checkbox in Setup for the Account object, and Salesforce automatically publishes a JSON payload every time an Account is created or updated. It's strictly for database synchronization." "Platform Events are custom. I define a custom object ending in __e. Then, inside an Apex trigger or Flow, I explicitly publish that event when a specific business rule is met."
Code: Publishing a Platform Event in Apex
// Assuming we created a Platform Event called 'Order_Ready__e'
List<Order_Ready__e> events = new List<Order_Ready__e>();
events.add(new Order_Ready__e(
    Order_Number__c = 'ORD-1001',
    Customer_Id__c = '001...'
));

// Publish the event to the Event Bus
List<Database.SaveResult> results = EventBus.publish(events);

for (Database.SaveResult sr : results) {
    if (sr.isSuccess()) {
        System.debug('Successfully published event.');
    }
}
