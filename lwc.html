Before You Start — Key Terms

If you're new to Salesforce or web development, learn these terms first. Every slide ahead uses them.

Web Development Terms

DOMDocument Object Model — the browser's internal tree of HTML elements. When you write <div>Hello</div>, the browser creates a DOM node you can interact with via JavaScript.
Shadow DOMA private, hidden DOM tree inside a component. CSS and JavaScript from outside CANNOT reach in. Like a room with locked doors — you can only access what's inside from inside.
ComponentA reusable piece of UI (HTML + JS + CSS bundled together). Think of it like a LEGO block — you build pages by combining components.
ReactiveWhen a variable's value changes, the UI automatically updates to show the new value. You don't manually refresh — the framework does it for you.
EventA message that says "something happened" (user clicked, data loaded, error occurred). Components talk to each other by sending and listening to events.
ModuleA JavaScript file that exports functions/classes. Other files can import from it. Like sharing a tool between team members.

Salesforce Terms

ApexSalesforce's server-side programming language (similar to Java). Runs on Salesforce servers. Used to query data, save records, and call external APIs.
SOQLSalesforce Object Query Language — like SQL but for Salesforce data. Example: SELECT Name FROM Account WHERE Id = '001XX'
DMLData Manipulation Language — operations that change data: INSERT, UPDATE, DELETE records in Salesforce database.
Governor LimitsSalesforce limits how much CPU, memory, and queries your code can use per transaction (shared server). Example: max 100 SOQL queries per transaction.
OrgA Salesforce organization/instance — your company's Salesforce environment. Like a workspace where all your data, code, and configuration lives.
SFDX / SF CLISalesforce command-line tool. Used to create projects, deploy code, run tests from your terminal (instead of clicking in browser).
Custom Metadata Type (CMT)Configuration records that control app behavior. Like a settings file — change configuration without changing code.
Lightning App BuilderDrag-and-drop page editor in Salesforce. You place your LWC components onto pages here — no code needed for page layout.

    }
}

Lifecycle Hooks

The Lifecycle Flow (in order of execution)

① constructor() — Fires first but @api properties are still undefined. Never use this for initialization that depends on parent-passed data.
② connectedCallback() ✅ — Component is now in the DOM. @api values are set by parent. This is where we initialize state, register document-level event listeners, and prefetch lookup data.
③ render() — LWC selects which HTML template to use. We don't override this (single template per component).
④ renderedCallback() — DOM is painted. Fires on EVERY re-render (not just first). We skip this because our UI is fully driven by reactive properties — no manual DOM manipulation needed.
⑤ disconnectedCallback() ✅ — Component is being removed from DOM. Critical for cleanup: remove document.addEventListener listeners to prevent memory leaks and ghost event handlers.

What We Use & Why

// connectedCallback — init state, register listeners
connectedCallback() {
    this._resetState();
    document.addEventListener('click', this._handler);
    this._prefetchResults(); // UX: preload data
}

// disconnectedCallback — prevent memory leaks
disconnectedCallback() {
    document.removeEventListener('click', this._handler);
}

🧠 Key Decisions Explained

Why NOT renderedCallback?
It fires on EVERY single re-render (data change, sort, filter, pagination). If we registered listeners here without a guard variable, we'd create hundreds of duplicate listeners. Since all our UI updates happen through reactive properties (LWC automatically re-renders the template when data changes), we never need to manually touch the DOM.

Why NOT errorCallback?
It's a last-resort boundary that only says "something broke" with no context. We prefer try/catch at the method level where we know exactly WHICH operation failed (was it the save? the lookup search? the model load?) — this lets us show specific, helpful error messages to the user.

Why prefetch in connectedCallback?
When a user clicks on a lookup field, they expect instant results. If we wait until click to fetch, there's a 300-500ms delay that feels sluggish. By silently fetching common results when the component mounts, the dropdown is already populated before the user even interacts with it.

            recordId: this._recordId,

  • Search: Need debounce + cancel control
  • Batch: Collect N IDs → 1 call (wire = N calls)
  • Sequence: Load → resolve names → render

Data Flow — @api Properties Down, Events Up

LWC enforces unidirectional data flow. Parent OWNS the data, passes it down via @api properties. Children can NEVER directly modify parent state — they must dispatch a CustomEvent asking the parent to make the change. This makes debugging predictable: you always know WHO changed WHAT and WHEN.

Parent (Orchestrator)
Owns ALL state: tabs[] array, parent{} header fields, errors{} per row. Calls Apex, processes results, pushes data down to children.
@api tab={data}
Child (EditableDataTable)
Receives the entire tab object via @api (rows, columns, config, search state). Renders the table UI. When user searches, sorts, or selects rows, it dispatches events UP to parent — never mutates the tab object directly.
CustomEvent('tabupdate')
Grandchild (InlineEditTable + LookupField)
Renders individual cells with inline editing. When a lookup field needs to search Apex, it emits a 'lookupsearch' event that BUBBLES up through the entire tree to the parent (bubbles:true, composed:true). The parent then calls Apex and pushes results back down via @api method.
☁️ Apex (Server)
ONLY parent talks to Apex.
getPageModel — load config
saveJson — persist changes
searchLookup — find records
batchNames — resolve IDs
@api method (Push Back)
Parent calls child's public method to push async results back:
receiveSearchResults(data)
clearErrors()
receiveLookupName(name)
Core Pattern: Data flows DOWN via @api properties (parent → child) | User actions flow UP via CustomEvent (child → parent) | Apex is centralized in parent ONLY — children request server data by emitting events, parent executes the call, then pushes results back via @api methods

Component Communication — 5 Patterns

LWC components communicate through a strict contract: data goes DOWN through properties, user actions go UP through events. This keeps components loosely coupled — a child doesn't know or care who its parent is.

① @api Properties Down — parent sets child's @api properties    ② Events Up — child dispatches CustomEvent, parent listens with on* handler    ③ @api Methods — parent calls child's public method directly    ④ Event Bubbling — grandchild event crosses shadow DOM to reach parent    ⑤ Apex Bridging — child requests data via event, parent calls Apex, pushes result back
Parent (TabParent)
Manages ALL state. Calls ALL Apex. Listens to 10+ child events. Pushes results back via @api methods.
Child (EditableDataTable)
Receives tab via @api. Dispatches tabupdate, rowaction, addrow, deleteselected events when user interacts.
Grandchild (LookupField — 3 levels deep)
Needs Apex search but can't import Apex directly. Dispatches 'lookupsearch' with bubbles:true + composed:true so event crosses all shadow DOM boundaries to reach parent.
↓ Pattern 1 — @api Properties Down: Parent passes data objects via @api: tab={data}, errors={errorObj}. Child reads but NEVER mutates these. One-way binding enforced by framework.
↑ Pattern 2 — Events Up: Child dispatches new CustomEvent('tabupdate', {detail: {searchTerm}}). Parent listens in template: ontabupdate={handleTabUpdate}. Loosely coupled — child doesn't know parent's handler name.
→ Pattern 3 — @api Methods: Parent queries child via this.template.querySelector('c-lookup-field').receiveSearchResults(data). Used when parent needs to PUSH async results back to child after Apex completes.
⬆ Pattern 4+5 — Bubbling + Apex Bridge: Grandchild emits event with {bubbles:true, composed:true} → crosses shadow DOM → parent catches it → calls Apex → pushes result back via @api method. No manual re-dispatch at each intermediate level.

Reactivity Traps — Common Mistakes

LWC's reactivity system only detects REFERENCE changes for objects/arrays. Mutating in-place = silent failure (UI won't update). These are the top 5 bugs intermediates hit.

❌ WRONG — These Silently Fail

// TRAP 1: Mutating array in place
this.rows.push(newRow);
// LWC sees: same reference → no re-render!

// TRAP 2: Changing object property directly
this.record.Name = 'Updated';
// LWC sees: same object reference → stale UI

// TRAP 3: Mutating @api property from child
this.tab.searchTerm = 'hello';
// ILLEGAL! @api is read-only in child
// Fails SILENTLY in production (no error!)

// TRAP 4: Array splice without reassignment
this.items.splice(2, 1);
// Array modified but LWC doesn't know

// TRAP 5: Nested object mutation
this.config.filters.active = true;
// Deep nested change = invisible to LWC

✅ CORRECT — New Reference = Re-render

// FIX 1: Spread into new array
this.rows = [...this.rows, newRow];
// New array reference → LWC re-renders!

// FIX 2: Spread into new object
this.record = { ...this.record, Name: 'Updated' };
// New object reference → template updates!

// FIX 3: Dispatch event to parent
this.dispatchEvent(new CustomEvent('search', {
    detail: { searchTerm: 'hello' }
}));
// Parent updates its state → passes new @api down

// FIX 4: Filter creates new array
this.items = this.items.filter((_, i) => i !== 2);
// New array → guaranteed re-render

// FIX 5: Deep clone + modify + reassign
const updated = JSON.parse(JSON.stringify(this.config));
updated.filters.active = true;
this.config = updated;
// Or: this.config = { ...this.config, 
//   filters: { ...this.config.filters, active: true }
// };

🧠 The Rule

Primitives (string, number, boolean): Always reactive. Just reassign: this.count = 5; → template updates automatically.

Objects/Arrays: Must create a NEW reference. LWC compares oldRef === newRef. If same reference → "nothing changed". Spread operator (...) is your best friend.

@api setter pattern: Use getter/setter when you need to react to parent-pushed changes: set tab(value) { this._tab = value; this._cache = null; }

Error Object Anatomy — All Shapes

Apex can throw errors in 4 different shapes depending on the context. Your error handler must check ALL of them.

Shape 1: AuraHandledException (most common)

// Apex throws this intentionally:
throw new AuraHandledException('Record not found');

// JS error object shape:
{
  body: {
    message: "Record not found"
  },
  ok: false,
  status: 400
}

// Access: error.body.message

Shape 2: DML Exception (save failure)

// Apex: insert fails due to validation rule

// JS error object shape:
{
  body: {
    output: {
      errors: [{
        message: "FIELD_CUSTOM_VALIDATION_EXCEPTION",
        errorCode: "FIELD_CUSTOM_VALIDATION_EXCEPTION"
      }],
      fieldErrors: {
        Amount__c: [{
          message: "Amount must be > 0"
        }]
      }
    }
  }
}

// Access: error.body.output.errors[0].message
// Or:     error.body.output.fieldErrors.Amount__c[0].message

Shape 3: Unhandled Apex Exception

// Apex: NullPointerException (unintentional)

// JS error object shape:
{
  body: {
    message: "An internal server error has occurred"
  },
  ok: false,
  status: 500
}

// In DEBUG mode you get more detail:
{
  body: {
    message: "List has no rows for assignment",
    stackTrace: "Class.MyCtrl.getRecord: line 15"
  }
}

// Access: error.body.message

Shape 4: Wire Service Error

// Wire adapter failure (no Apex involved)

// JS error object shape:
{
  body: {
    message: "An error occurred while ...",
  },
  // Sometimes it's an array:
  body: [{ message: "...", errorCode: "..." }]
}

// Access: Array.isArray(error.body) 
//   ? error.body[0].message 
//   : error.body.message

Universal Handler (covers ALL shapes)

export function reduceErrors(error) {
  if (!error) return ['Unknown error'];
  if (typeof error === 'string') return [error];
  if (Array.isArray(error.body)) 
    return error.body.map(e => e.message);
  if (error.body?.output?.errors)
    return error.body.output.errors.map(e => e.message);
  if (error.body?.message) 
    return [error.body.message];
  if (error.message) return [error.message];
  return ['Unknown error'];
}

Anti-Patterns — What NOT to Do

These patterns WORK (code runs) but cause bugs, performance issues, or maintenance nightmares. Avoid them.

🚫 Architecture Anti-Patterns

  • Apex in every child
    ❌ Each child imports its own Apex methods
    ✅ Centralize in parent. Children request via events.
    Why: Duplicate error handling, impossible to coordinate loading states, harder to test.
  • God component (1000+ lines)
    ❌ One component does everything
    ✅ Split by responsibility (table, modal, validator)
    Why: Impossible to unit test, can't reuse pieces, merge conflicts in team.
  • Storing state in DOM
    ❌ Reading values from this.template.querySelector('input').value
    ✅ Bind to reactive properties: value={myProp}
    Why: DOM reads are slow, can return stale values, not reactive.
  • Hardcoded field names
    ❌ if (record.Status__c === 'Closed')
    ✅ Use Custom Metadata/Custom Labels for config
    Why: Fails across orgs, breaks on field rename, not admin-configurable.

🚫 Performance Anti-Patterns

  • Apex call in renderedCallback
    ❌ Make server call every render cycle
    ✅ Call in connectedCallback or event handler
    Why: renderedCallback fires on EVERY data change. You'll make 100s of calls as user types.
  • No debounce on search
    ❌ Call Apex on every keystroke
    ✅ Wait 300ms after last keystroke
    Why: "Testing" = 7 Apex calls instead of 1. Hits governor limits, shows flickering results.
  • Getters with heavy logic
    ❌ get processedRows() { return rows.map(...).filter(...).sort(...); }
    ✅ Cache result, invalidate on data change
    Why: Getters run on EVERY template access (multiple times per render cycle).
  • setTimeout for timing
    ❌ setTimeout(() => this.doStuff(), 500)
    ✅ Use await Promise.resolve() or proper lifecycle
    Why: Race conditions, untestable, arbitrary delays that work on fast networks but fail on slow.

🚫 Event & Data Anti-Patterns

  • Always bubbles:true
    ❌ Every event set to bubble
    ✅ Only bubble when crossing 2+ levels
    Why: Parent AND grandparent both catch it. Accidental double-handling, hard to trace which listener fired.
  • Mutating event.detail
    ❌ const data = event.detail; data.name = 'new';
    ✅ Clone first: const data = {...event.detail};
    Why: event.detail is shared reference. Mutating it affects the sender's data.
  • Missing key in for:each
    ❌ key={index} or missing key
    ✅ key={record.Id} (unique stable identifier)
    Why: Without stable key, LWC destroys/recreates ALL items on any change. Causes: lost focus, lost scroll position, lost input state.
  • Two-way binding hack
    ❌ Child modifies @api property internally
    ✅ Emit event, let parent update, receive new @api
    Why: Works in dev but fails unpredictably in production. LWC may overwrite your change on next render cycle.

Performance & Best Practices

These patterns solve real production problems: pages with 500+ rows, 50+ lookup fields, users on slow networks, and long editing sessions that accumulate memory.

⚡ Performance

  • LRU Cache
    Least-Recently-Used eviction. Lookup results cached (max 50 entries shared across ALL instances). Cell render cache (max 500). Modal search cache (max 20 pages). Prevents redundant computations and Apex calls.
  • Debounce (300ms)
    User types "Test" = 4 keystrokes. Without debounce = 4 Apex calls. With debounce = waits 300ms after LAST keystroke → 1 Apex call. Saves governor limits and prevents UI flicker.
  • Batch Apex Calls
    Page has 50 lookup fields with IDs. Instead of 50 parallel getLookupName() calls (hits concurrent limit), we collect all IDs and make 1 batchGetLookupNames() call. Returns all names in one response.
  • Prefetch on Mount
    LookupField silently fetches initial results in connectedCallback. When user clicks the field, dropdown is already populated — zero perceived delay. Cache is shared so other instances benefit too.
  • Computed Cache with Invalidate-on-Set
    Expensive getters (paginatedRows, processedColumns) compute ONCE then cache. When @api setter fires, cache is set to null. Next render recomputes from fresh data. Avoids recalculating on every template access.

🏗️ Architecture

  • Single Responsibility
    Each component does exactly ONE thing. Table renders rows. Lookup handles search. Modal handles form. Validator checks rules. This makes each piece testable, reusable, and debuggable in isolation.
  • Apex Centralized in Parent
    ONLY TabParent imports Apex methods. Children request data via events. This means: one place for error handling, one place for loading states, no duplicate imports, easier governor limit tracking.
  • Metadata-Driven (Custom Metadata Types)
    Adding a new tab = create a CMT record in Setup (no code, no deployment). Change field order = edit CMT. Admins can configure without developers. Reduces change cycle from 2 weeks to 5 minutes.
  • Immutable Updates
    LWC detects changes by REFERENCE comparison. If you mutate an array in-place (arr[0].value = x), LWC doesn't know it changed → stale UI. Always create new array/object: this.fields = [...updated]. Guaranteed re-render.
  • Barrel Module Pattern
    rdFsDynamicTableUtils re-exports functions from 3 sub-modules. Consumers write: import { fn } from 'c/rdFsDynamicTableUtils'. If we refactor sub-modules internally, zero consumer changes needed.

🛡️ Memory & UX

  • Listener Cleanup
    Document-level listeners (document.addEventListener) are NOT cleaned up by LWC garbage collection. We MUST remove them in disconnectedCallback or they persist as ghost handlers — firing on removed components and leaking memory.
  • Cache Eviction (LRU)
    Every cache has a MAX size. When full, the Least Recently Used entry is removed. Without this: 30-minute editing session → megabytes of stale search results in memory → browser tab slows down or crashes.
  • Loading Spinners
    Every async operation (Apex call, lookup search) shows lightning-spinner immediately. User knows something is happening. Without spinners: user clicks Save, nothing visible happens for 2 seconds, they click again → duplicate DML.
  • Confirmation Dialogs
    LightningConfirm.open() before destructive actions (unsaved changes on refresh, bulk delete). Prevents accidental data loss. Async/await pattern makes the code sequential and readable.
  • Sort Persistence
    User sorts column A descending, then paginates to page 3. Sort state is stored in the tab object and preserved. Without this: every pagination reset would lose the sort → frustrating UX.

Step 7: Draft State Management

// Track ALL changes in a Map (never mutate original data)
_drafts = new Map();
// Structure: Map<rowKey, Map<fieldName, newValue>>

// When user edits a cell:
handleCellChange(event) {
    const { rowKey, field } = event.currentTarget.dataset;
    const value = event.detail.value;
    
    if (!this._drafts.has(rowKey)) {
        this._drafts.set(rowKey, new Map());
    }
    this._drafts.get(rowKey).set(field, value);
    
    // Mark cell as dirty (green highlight)
    this._rebuildRow(rowKey);
    // Emit to parent so it can track unsaved changes
    this.dispatchEvent(new CustomEvent('draftsave', {
        detail: { rowKey, field, value }
    }));
}

// Check if a cell is dirty:
_isDirty(rowKey, field) {
    return this._drafts.has(rowKey) 
        && this._drafts.get(rowKey).has(field);
}

// Get current value (draft overrides original):
_getCurrentValue(rowKey, field, originalValue) {
    if (this._isDirty(rowKey, field)) {
        return this._drafts.get(rowKey).get(field);
    }
    return originalValue;
}

Step 9: Save Serialization (→ Apex)

// Serialize ALL changes into JSON for saveJson() Apex method
_buildSavePayload() {
    const payload = {
        newRecords: [],
        updatedRecords: [],
        deletedIds: [...this._deletedKeys]
    };
    
    // New rows (have temp keys, no Salesforce Id)
    this._rows.filter(r => r.isNew).forEach(row => {
        const record = {};
        row.cells.forEach(cell => {
            record[cell.field] = this._getCurrentValue(
                row.key, cell.field, cell.value);
        });
        payload.newRecords.push(record);
    });
    
    // Modified existing rows (have real Ids + draft changes)
    this._drafts.forEach((fieldMap, rowKey) => {
        if (rowKey.startsWith('NEW_')) return; // Skip new
        const record = { Id: rowKey };
        fieldMap.forEach((value, field) => {
            record[field] = value;
        });
        payload.updatedRecords.push(record);
    });
    
    return JSON.stringify(payload);
}

// Call Apex:
async handleSave() {
    const json = this._buildSavePayload();
    try {
        await saveJson({ recordId: this._recordId, 
                         tabsJson: json });
        this._showToast('Success', 'Records saved', 'success');
        this._clearDraftState();
    } catch (error) {
        this._showToast('Error', formatApexError(error), 'error');
    }
}

Shadow DOM & CSS Isolation

Every LWC component has its own Shadow DOM boundary. CSS styles defined in one component CANNOT leak into or be affected by another component. This is a core interview topic.

What is Shadow DOM?

Each LWC component renders inside an isolated DOM subtree. The browser enforces encapsulation:

  • CSS Scoping — Styles in component A can NEVER affect component B's elements. No global CSS bleed.
  • DOM Encapsulation — document.querySelector() from outside CANNOT reach inside a component's shadow tree.
  • this.template.querySelector() — The ONLY way to access elements inside your own component.
  • Event Retargeting — Events that cross shadow boundary get their target retargeted to the host element (so parent can't see child's internal structure).

🧠 Interview: "How do you style a child component from parent?"

Answer: You CAN'T directly. Options:

  • CSS Custom Properties (variables) — Parent sets --my-color: red; child uses var(--my-color). Variables pierce shadow DOM.
  • SLDS Design Tokens — Use standard Salesforce tokens that work everywhere.
  • @api className — Pass a class name as @api property, child applies it internally.
  • :host selector — Style the component's outer host element from within.

Performance Profiling — Measuring What Matters

Senior devs don't guess — they measure. Here's how to find and fix render bottlenecks in production-scale LWC apps.

Chrome Performance Tab — Flame Graph

  • Record — Click ⏺ → perform the slow action → stop. Look for long yellow (JS) bars.
  • LWC markers — In the flame chart, LWC adds markers like lwc-render, lwc-rehydrate. Filter by "lwc" to isolate framework time vs your code time.
  • Identify re-render storms — If you see 50+ consecutive lwc-render calls, a reactive property is being set in a loop (common with array.forEach that assigns this.data multiple times).
  • Layout thrashing — If you see purple "Layout" blocks between JS, you're reading DOM geometry (offsetWidth) then writing (style.width) in a loop. Batch all reads, then all writes.

  • Batch reactive assignments — Don't set 5 properties in sequence (5 renders). Build object, assign once: Object.assign(this, computed);

  • @api properties — Dynamically rendered component receives @api props via the template attributes. You can't call @api methods until after render (use renderedCallback guard).
  • Event listening — Events from dynamic component bubble normally. Parent listens with standard on* handlers in template.

_trapFocus(event) {

        'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'

Infinite Scroll Pattern

offset = 0;
pageSize = 50;
allData = [];
isLoading = false;
hasMore = true;

async connectedCallback() {
  await this.loadMore();
}

async loadMore() {
  if (this.isLoading || !this.hasMore) return;
  this.isLoading = true;
  try {
    const records = await getRecords({
      offset: this.offset,
      pageSize: this.pageSize
    });
    this.allData = [...this.allData, ...records];
    this.offset += records.length;
    this.hasMore = records.length === this.pageSize;
  } finally {
    this.isLoading = false;
  }
}

handleScroll(event) {
  const { scrollTop, scrollHeight, clientHeight }
    = event.target;
  if (scrollHeight - scrollTop - clientHeight < 100) {
    this.loadMore(); // Near bottom → load more
  }
}

export class LocalStorageAdapter {
  connected = false;
  key;

  constructor(dataCallback) {
    this.dataCallback = dataCallback;
  }

  connect() { this.connected = true; this.fetch(); }
  disconnect() { this.connected = false; }

    }
  }
}

Advanced Error Handling & Retry Patterns

Error Reduction Utility

// errorUtils.js — reusable error parser
export function reduceErrors(errors) {
  if (!Array.isArray(errors)) errors = [errors];
  return errors
    .filter(e => !!e)
    .map(e => {
      // UI API errors
      if (Array.isArray(e.body)) {
        return e.body.map(b => b.message);
      }
      // Apex errors
      if (e.body && e.body.message) {
        return e.body.message;
      }
      // Page-level errors
      if (e.body && e.body.pageErrors) {
        return e.body.pageErrors.map(p => p.message);
      }
      // Field-level errors
      if (e.body && e.body.fieldErrors) {
        return Object.values(e.body.fieldErrors)
          .flat().map(f => f.message);
      }
      // JS errors
      if (typeof e.message === 'string') return e.message;
      return e.statusText || 'Unknown error';
    })
    .flat();
}

Retry & Timeout Pattern

// Retry with exponential backoff
async callWithRetry(fn, maxRetries = 3) {
  for (let i = 0; i <= maxRetries; i++) {
    try {
      return await fn();
    } catch (error) {
      if (i === maxRetries) throw error;
      const delay = Math.pow(2, i) * 1000; // 1s, 2s, 4s
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

// Usage:
async handleSave() {
  try {
    const result = await this.callWithRetry(
      () => saveRecord({ record: this.record })
    );
    this.showToast('Saved', 'success');
  } catch (error) {
    this.errors = reduceErrors(error);
    this.showToast(this.errors[0], 'error');
  }
}

Common Questions — Quick Answers (Part 1: Fundamentals)

Rapid-fire Q&A covering LWC fundamentals, decorators, lifecycle, and rendering.

QuestionAnswer
What's the difference between Aura and LWC?LWC uses native Web Components (Shadow DOM, custom elements), standard JavaScript ES6+. Aura uses a proprietary framework with its own rendering engine, event model ({!v.attr}), and helper files. LWC is 3x faster (no framework overhead), smaller bundles, uses ES modules, and has Jest testing support.
What are the 4 files in an LWC component?.html (template), .js (logic/state), .css (scoped styles), .js-meta.xml (configuration — targets, API version, design attributes). Only .js and .js-meta.xml are required. HTML and CSS are optional (service components have no HTML).
What is the LWC reactivity model?ONE-WAY data binding. Primitives are reactive by default (reassign to trigger re-render). Objects/arrays require NEW reference (spread operator). No two-way binding like Aura. Template re-renders automatically when any reactive property changes.
Explain the 3 decorators: @api, @wire, @track@api = public property/method (settable by parent). @wire = reactive data fetching (auto-fires when params change). @track = DEPRECATED since Spring '20 (all fields are tracked by default now). Only use @api and @wire in new code.
Why is @track deprecated?Before Spring '20, only @track fields triggered re-render. After Spring '20, ALL class fields are reactive by default. @track is redundant — it still works but adds no value. The only caveat: deep object mutation still requires new reference regardless.
What is @api getter/setter pattern?Intercept property changes from parent: set tab(value) { this._tab = value; this._invalidateCache(); }. Use when you need side effects on property change (cache invalidation, data transformation, validation).
What are lifecycle hooks in LWC? List in order.constructor() → connectedCallback() → render() → renderedCallback() → disconnectedCallback(). Also: errorCallback(error, stack) for error boundaries. Most used: connectedCallback (init) and disconnectedCallback (cleanup).
When does renderedCallback fire?After EVERY render (not just first). Fires when any reactive property changes. Dangerous for Apex calls or addEventListener (creates duplicates). Use a guard: if (this._hasRendered) return; this._hasRendered = true;
What is errorCallback?Fires when a child component throws during rendering. Receives (error, stack). Used as an error boundary — parent catches child's render error and shows fallback UI. Does NOT catch errors in event handlers or async code.
What are lwc:if, lwc:elseif, lwc:else?Modern conditional directives (replaced if:true/if:false). Guarantee mutual exclusivity — only one branch renders. Applied to <template> tags. lwc:if={condition} evaluates truthy/falsy.
Why was if:true/if:false deprecated?No mutual exclusivity guarantee (both could render simultaneously in edge cases). No else-if chaining. lwc:if/elseif/else is cleaner syntax and guarantees only one branch is in DOM at a time.
What is for:each and what is the key?Iterates over arrays: for:each={items} for:item="item". key={item.id} is REQUIRED — must be a unique stable identifier (NOT array index). Key enables efficient DOM diffing (only changed items re-render).
Why NOT use index as key?If you delete item at index 2, all items after it shift. LWC thinks items 2,3,4,5... ALL changed (new key mapping) → destroys and recreates every DOM element. With stable IDs, only the deleted one is removed.
What is iterator:it vs for:each?iterator provides {value, index, first, last} properties. Useful for special styling (first/last item). for:each is simpler when you don't need first/last detection. Both work — for:each is more common.
How does LWC template rendering work internally?LWC uses a virtual DOM diffing algorithm. When state changes, it creates a new virtual tree, diffs against the old one, and applies minimal DOM mutations. This is why new references are needed — same reference = "no diff needed".

Q1. Explain the lifecycle hooks in LWC with an example.

Answer: LWC lifecycle hooks are callback methods that fire at specific points in a component's life. They execute in this order:

// 1. constructor() — Component instance created. DO NOT access DOM here.
constructor() {
    super(); // MUST call super() first
    this.greeting = 'Hello'; // Only set primitive defaults
}

// 2. connectedCallback() — Component inserted into DOM. @api properties are available.
connectedCallback() {
    this.loadInitialData();
    document.addEventListener('keydown', this._handleKeydown);
}

// 3. renderedCallback() — DOM has been painted. Fires on EVERY re-render.
renderedCallback() {
    if (this._chartInitialized) return; // Guard against repeated execution
    this._chartInitialized = true;
    this._initChart(); // Safe to access this.template.querySelector()
}

// 4. disconnectedCallback() — Component removed from DOM. Cleanup listeners.
disconnectedCallback() {
    document.removeEventListener('keydown', this._handleKeydown);
}

// 5. errorCallback(error, stack) — Child component throws unhandled error.
errorCallback(error, stack) {
    this.errorMessage = error.message;
    console.error('Child error:', stack);
}

Key rules: Use connectedCallback for initialization (not constructor). Use renderedCallback with a guard flag to avoid infinite loops. Always clean up in disconnectedCallback.

Q3. How does data binding work in LWC?

Answer: LWC uses one-way data binding (unlike Aura's two-way). Data flows from JS → template only. The template CANNOT directly modify JS properties.

// JS property (reactive by default since Spring '20)
export default class MyComp extends LightningElement {
    greeting = 'World'; // Reactive — template auto-updates when this changes

    handleInput(event) {
        // Template cannot update greeting directly
        // You MUST update it in JS via event handler
        this.greeting = event.target.value; // This triggers re-render
    }
}

// Template binds with {propertyName}:
// <p>Hello, {greeting}!</p>
// <lightning-input value={greeting} onchange={handleInput}>

// WHY one-way? 
// - Predictable: you always know WHERE data changed (in the handler)
// - Debuggable: set breakpoint in handler, trace every change
// - No accidental mutations from template side

Key point: All class properties are reactive by default. When you reassign them (this.x = newValue), the template re-renders. For objects/arrays, you must create a NEW reference (spread operator) — mutation won't trigger re-render.

Q9. Explain Lightning Message Service (LMS) in LWC.

Answer: LMS enables communication between components that have NO parent-child relationship — siblings, components on different page regions, even Aura ↔ LWC ↔ Visualforce.

// Step 1: Create Message Channel (metadata file)
// force-app/main/default/messageChannels/RecordSelected__c.messageChannel-meta.xml
<?xml version="1.0"?>
<LightningMessageChannel xmlns="http://soap.sforce.com/2006/04/metadata">
    <masterLabel>Record Selected</masterLabel>
    <isExposed>true</isExposed>
    <lightningMessageFields>
        <fieldName>recordId</fieldName>
        <fieldName>recordName</fieldName>
    </lightningMessageFields>
</LightningMessageChannel>

// Step 2: PUBLISHER component (sends the message)
import { publish, MessageContext } from 'lightning/messageService';
import RECORD_SELECTED from '@salesforce/messageChannel/RecordSelected__c';

@wire(MessageContext) messageContext;

handleRecordClick(event) {
    publish(this.messageContext, RECORD_SELECTED, {
        recordId: event.currentTarget.dataset.id,
        recordName: event.currentTarget.dataset.name
    });
}

// Step 3: SUBSCRIBER component (receives the message — anywhere on the page)
import { subscribe, unsubscribe, MessageContext } from 'lightning/messageService';
import RECORD_SELECTED from '@salesforce/messageChannel/RecordSelected__c';

@wire(MessageContext) messageContext;
_subscription = null;

connectedCallback() {
    this._subscription = subscribe(this.messageContext, RECORD_SELECTED, (message) => {
        this.selectedId = message.recordId;
        this.selectedName = message.recordName;
        this.loadRecordDetails();
    });
}

disconnectedCallback() {
    unsubscribe(this._subscription);
    this._subscription = null;
}

Use cases: Utility bar ↔ record page, list component ↔ detail component on same page, filter panel ↔ data grid.

Q13. Custom datatable with inline editing — how to handle field validation?

Answer: Validate at two layers: (1) Cell-level instant feedback, (2) Row-level before save.

// ═══ Layer 1: Cell-level validation (instant) ═══
handleCellChange(event) {
    const { rowKey, field } = event.currentTarget.dataset;
    const value = event.detail.value;
    const column = this._columns.find(c => c.fieldName === field);
    
    // Validate based on column config
    const error = this._validateCell(value, column);
    if (error) {
        event.target.setCustomValidity(error);  // Shows red border + message
        event.target.reportValidity();
        return; // Don't save invalid value to draft
    }
    event.target.setCustomValidity(''); // Clear previous error
    this._saveToDraft(rowKey, field, value);
}

_validateCell(value, column) {
    if (column.required && (!value || value.trim() === '')) 
        return `${column.label} is required`;
    if (column.type === 'NUMBER' && isNaN(value)) 
        return 'Must be a number';
    if (column.type === 'NUMBER' && column.min != null && Number(value) < column.min) 
        return `Minimum value is ${column.min}`;
    if (column.maxLength && value?.length > column.maxLength) 
        return `Maximum ${column.maxLength} characters`;
    return null; // Valid
}

// ═══ Layer 2: Row-level validation (before save) ═══
_validateAllRows() {
    const errors = {};
    this._rows.forEach(row => {
        const rowErrors = [];
        this._columns.forEach(col => {
            const value = this._getCurrentValue(row.key, col.fieldName, /* original */);
            const error = this._validateCell(value, col);
            if (error) rowErrors.push({ field: col.fieldName, message: error });
        });
        // Cross-field validation
        if (row.startDate && row.endDate && row.startDate > row.endDate)
            rowErrors.push({ field: 'EndDate', message: 'End date must be after start date' });
        if (rowErrors.length) errors[row.key] = rowErrors;
    });
    return errors; // Empty = valid
}

Q18. Data not displaying after render but updates after user interaction — what's the issue?

Answer: This is almost always a reactivity issue — you're mutating an object/array in place instead of creating a new reference.

// ❌ THE BUG — mutation doesn't trigger re-render:
async loadData() {
    const result = await getRecords();
    this.records.push(...result);  // WRONG! Same array reference
    // LWC sees: same reference → "nothing changed" → no re-render
    // But when user clicks something → triggers OTHER reactive update → NOW it shows
}

// ✅ THE FIX — create new reference:
async loadData() {
    const result = await getRecords();
    this.records = [...this.records, ...result]; // NEW array reference → re-renders!
}

Q20. LWC interacts with a third-party API — how to handle errors during the API call?

Answer: Third-party API calls must go through Apex (LWC cannot make direct HTTP callouts due to CORS/security). Handle errors at both layers:

// ═══ APEX: Make the callout + wrap errors ═══
@AuraEnabled
public static String getExternalData(String endpoint) {
    try {
        HttpRequest req = new HttpRequest();
        req.setEndpoint(endpoint);
        req.setMethod('GET');
        req.setTimeout(30000); // 30 sec timeout
        req.setHeader('Authorization', 'Bearer ' + getToken());
        
        HttpResponse res = new Http().send(req);
        
        if (res.getStatusCode() == 200) {
            return res.getBody();
        } else if (res.getStatusCode() == 401) {
            throw new AuraHandledException('Authentication failed. Token expired.');
        } else if (res.getStatusCode() == 429) {
            throw new AuraHandledException('API rate limit exceeded. Try again in 60 seconds.');
        } else {
            throw new AuraHandledException('API Error: ' + res.getStatus() + ' (' + res.getStatusCode() + ')');
        }
    } catch (System.CalloutException e) {
        // Timeout, DNS failure, connection refused
        throw new AuraHandledException('Service unavailable: ' + e.getMessage());
    }
}

// ═══ LWC: Handle the response + show user-friendly messages ═══
import getExternalData from '@salesforce/apex/ExternalApiCtrl.getExternalData';

async fetchApiData() {
    this.isLoading = true;
    this.errorMessage = null;
    
    try {
        const rawJson = await getExternalData({ endpoint: this._apiUrl });
        this.apiData = JSON.parse(rawJson);
    } catch (error) {
        // Categorize error for UX
        const msg = error.body?.message || 'Unknown error';
        
        if (msg.includes('rate limit')) {
            this.errorMessage = '⏳ Too many requests. Please wait 60 seconds.';
            this._scheduleRetry(60000); // Auto-retry after 1 min
        } else if (msg.includes('Authentication')) {
            this.errorMessage = '🔐 Session expired. Please refresh the page.';
        } else if (msg.includes('unavailable')) {
            this.errorMessage = '🔌 External service is down. Try again later.';
        } else {
            this.errorMessage = `❌ ${msg}`;
        }
        
        this.showToast('API Error', this.errorMessage, 'error', 'sticky');
    } finally {
        this.isLoading = false;
    }
}

// Retry pattern
_scheduleRetry(delayMs) {
    this._retryTimer = setTimeout(() => this.fetchApiData(), delayMs);
}
disconnectedCallback() {
    clearTimeout(this._retryTimer); // Cleanup
}

      <lightning-input type="search" label={label}
        value={searchTerm} onchange={handleSearch}

      </lightning-input>

                <span class="slds-truncate">{record.Name}</span>
                <span class="slds-text-body_small">{record.subtitle}</span>
              </li>

            </template>
          </ul>
        </div>
      </template>
    </template>
  </div>

// ═══ customLookup.js ═══
import { LightningElement, api } from 'lwc';
import search from '@salesforce/apex/LookupCtrl.search';

export default class CustomLookup extends LightningElement {
    @api label = 'Lookup';
    @api objectName = 'Account';
    @api iconName = 'standard:account';
    @api value; // Pre-selected record Id
    
    searchTerm = '';
    results = [];
    selectedRecord = null;
    showDropdown = false;
    _timer;
    
    // Debounced search
    handleSearch(event) {
        this.searchTerm = event.target.value;
        clearTimeout(this._timer);
        if (this.searchTerm.length < 2) { this.showDropdown = false; return; }
        this._timer = setTimeout(() => this._doSearch(), 300);
    }
    
    async _doSearch() {
        try {
            this.results = await search({ 
                objectName: this.objectName, 
                searchTerm: this.searchTerm 
            });
            this.showDropdown = true;
        } catch (e) { console.error(e); }
    }
    
    handleSelect(event) {
        const id = event.currentTarget.dataset.id;
        this.selectedRecord = this.results.find(r => r.Id === id);
        this.showDropdown = false;
        this.searchTerm = '';
        // Notify parent of selection
        this.dispatchEvent(new CustomEvent('change', { detail: { value: id, record: this.selectedRecord } }));
    }
    
    handleClear() {
        this.selectedRecord = null;
        this.dispatchEvent(new CustomEvent('change', { detail: { value: null, record: null } }));
    }
    
    get noResults() { return this.results.length === 0 && this.searchTerm.length >= 2; }
}
