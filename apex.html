SOQL & SOSL — Querying Data

SOQL = Salesforce Object Query Language (like SQL for one object + relationships). SOSL = Salesforce Object Search Language (full-text search across multiple objects).

SOQL Fundamentals

// Basic query
List<Account> accts = [
    SELECT Id, Name, Industry, AnnualRevenue
    FROM Account
    WHERE Industry = 'Technology'
    AND AnnualRevenue > 1000000
    ORDER BY Name ASC
    LIMIT 50
    OFFSET 10
];

// Relationship queries (parent → child)
List<Account> withContacts = [
    SELECT Name, 
        (SELECT FirstName, LastName, Email
         FROM Contacts
         WHERE IsActive__c = true)
    FROM Account
    WHERE Id IN :accountIds
];

// Child → parent (dot notation)
List<Contact> contacts = [
    SELECT FirstName, Account.Name, Account.Industry
    FROM Contact
    WHERE Account.Industry = 'Tech'
];

// Aggregate queries
List<AggregateResult> results = [
    SELECT Industry, COUNT(Id) cnt, SUM(AnnualRevenue) total
    FROM Account
    GROUP BY Industry
    HAVING COUNT(Id) > 5
];

SOSL — Full-Text Search

// Search across multiple objects at once
List<List<SObject>> results = [
    FIND 'Acme*' IN ALL FIELDS
    RETURNING 
        Account(Name, Industry WHERE Industry = 'Tech'),
        Contact(FirstName, LastName, Email),
        Opportunity(Name, Amount)
    LIMIT 20
];
List<Account> accts = (List<Account>) results[0];
List<Contact> contacts = (List<Contact>) results[1];

Dynamic SOQL (Safe)

// Build queries dynamically
String objectName = 'Account';
String fields = 'Id, Name, Industry';
String conditions = 'Industry = :industry';

String query = 'SELECT ' + fields 
    + ' FROM ' + objectName 
    + ' WHERE ' + conditions;

// ⚠️ SECURITY: Use bind variables, not concatenation!
// ✅ Safe: WHERE Name = :userInput
// ❌ Unsafe: WHERE Name = \'' + userInput + '\''
// ❌ = SQL Injection vulnerability!

List<Account> results = Database.query(query);

DML Operations — Creating, Updating, Deleting Records

DML (Data Manipulation Language) statements persist changes to the database. Always bulkify — operate on lists, not single records.

6 DML Statements

// INSERT — create new records
Account acc = new Account(Name = 'New Corp');
insert acc;
// acc.Id is now populated!

// UPDATE — modify existing records
acc.Industry = 'Technology';
update acc;

// UPSERT — insert or update based on external ID
Account[] accts = getExternalData();
upsert accts External_Id__c; // Match on this field

// DELETE — move to Recycle Bin
delete acc;

// UNDELETE — restore from Recycle Bin
undelete acc;

// MERGE — combine duplicate records
Account master = [SELECT Id FROM Account WHERE Name = 'Acme'];
Account dupe = [SELECT Id FROM Account WHERE Name = 'ACME Corp'];
merge master dupe;
// dupe's children re-parented to master

Bulkification Pattern

// ❌ BAD — DML inside loop (governor limit!)
for (Contact c : contacts) {
    c.Status__c = 'Active';
    update c; // 150 DML limit hit at record 150!
}

// ✅ GOOD — Collect and execute once
for (Contact c : contacts) {
    c.Status__c = 'Active';
}
update contacts; // ONE DML for all records

Database Methods (Partial Success)

// Database.insert allows partial success
Database.SaveResult[] results = Database.insert(
    accounts, false // allOrNone = false
);

for (Database.SaveResult sr : results) {
    if (sr.isSuccess()) {
        System.debug('Inserted: ' + sr.getId());
    } else {
        for (Database.Error err : sr.getErrors()) {
            System.debug('Error: ' + err.getMessage());
            System.debug('Fields: ' + err.getFields());
        }
    }
}

// vs. standard DML (all or nothing):
// insert accounts; ← throws exception if ANY fail

Triggers — Automating on Record Changes

Triggers execute Apex code before or after DML events on records. Always use a handler pattern — never put logic directly in the trigger.

Trigger Syntax & Events

// AccountTrigger.trigger
trigger AccountTrigger on Account (
    before insert, before update, before delete,
    after insert, after update, after delete, 
    after undelete
) {
    AccountTriggerHandler handler = new AccountTriggerHandler();
    
    if (Trigger.isBefore) {
        if (Trigger.isInsert) handler.beforeInsert(Trigger.new);
        if (Trigger.isUpdate) handler.beforeUpdate(Trigger.new, Trigger.oldMap);
        if (Trigger.isDelete) handler.beforeDelete(Trigger.old);
    }
    if (Trigger.isAfter) {
        if (Trigger.isInsert) handler.afterInsert(Trigger.new);
        if (Trigger.isUpdate) handler.afterUpdate(Trigger.new, Trigger.oldMap);
        if (Trigger.isDelete) handler.afterDelete(Trigger.old);
    }
}

Trigger Context Variables

VariableDescription
Trigger.newList of NEW records (insert/update)
Trigger.oldList of OLD records (update/delete)
Trigger.newMapMap<Id, SObject> of new records
Trigger.oldMapMap<Id, SObject> of old records
Trigger.isInsertBoolean — is this an insert?
Trigger.isBeforeBoolean — is this before DML?
Trigger.sizeTotal records in trigger batch

Handler Pattern (Best Practice)

public class AccountTriggerHandler {
    
    public void beforeUpdate(List<Account> newAccs, 
                             Map<Id, Account> oldMap) {
        List<Account> changedIndustry = new List<Account>();
        
        for (Account acc : newAccs) {
            Account old = oldMap.get(acc.Id);
            // Only process records where field actually changed
            if (acc.Industry != old.Industry) {
                changedIndustry.add(acc);
            }
        }
        
        if (!changedIndustry.isEmpty()) {
            updateRelatedContacts(changedIndustry);
        }
    }
    
    private void updateRelatedContacts(List<Account> accs) {
        // Bulkified helper logic
        List<Contact> toUpdate = [
            SELECT Id, Account.Industry 
            FROM Contact 
            WHERE AccountId IN :accs
        ];
        for (Contact c : toUpdate) {
            c.Industry__c = c.Account.Industry;
        }
        update toUpdate;
    }
}

Before vs After — When to Use

Use BeforeUse After
Validate field valuesUpdate related/child records
Set default valuesSend email notifications
Modify same record (no DML needed!)Create audit log records
Prevent save with addError()Call external systems

Rule: Use before to modify the SAME record (it's free — no extra DML). Use after when you need the Id (insert) or need to modify OTHER records.

Governor Limits — The #1 Interview Topic

Salesforce is multi-tenant — one org's bad code can't bring down others. Governor limits enforce fair resource usage per transaction.

Critical Limits (Synchronous)

ResourceLimit
SOQL queries100 per transaction
SOQL rows returned50,000 rows
DML statements150 per transaction
DML rows10,000 records
Callouts100 per transaction
CPU time10,000 ms
Heap size6 MB (sync) / 12 MB (async)
Future calls50 per transaction

How to Avoid Hitting Limits

// ❌ SOQL inside loop (100 limit)
for (Account acc : accounts) {
    List<Contact> cons = [SELECT Id FROM Contact 
                          WHERE AccountId = :acc.Id];
}

// ✅ Query ONCE, use Map to organize
Map<Id, List<Contact>> contactsByAcct = new Map<Id, List<Contact>>();
for (Contact c : [SELECT Id, AccountId FROM Contact
                  WHERE AccountId IN :accounts]) {
    if (!contactsByAcct.containsKey(c.AccountId)) {
        contactsByAcct.put(c.AccountId, new List<Contact>());
    }
    contactsByAcct.get(c.AccountId).add(c);
}

// ❌ DML inside loop (150 limit)
for (Contact c : contacts) { update c; }

// ✅ Collect all, DML once
update contacts;

// ❌ Multiple queries for same data
Account a = [SELECT Name FROM Account WHERE Id = :id];
Account b = [SELECT Industry FROM Account WHERE Id = :id];

// ✅ One query, all fields
Account a = [SELECT Name, Industry FROM Account WHERE Id = :id];

Collections — List, Set, Map

Collections are the foundation of bulkified Apex. Master these patterns and you'll never hit governor limits.

List — Ordered, Allows Duplicates

List<String> names = new List<String>();
names.add('Alice');
names.add('Bob');
names.add('Alice'); // Allowed!
System.debug(names[0]); // 'Alice'
System.debug(names.size()); // 3

// SOQL returns a List

// Common patterns
names.sort(); // Alphabetical
names.contains('Alice'); // true
names.remove(0); // Remove by index
List<String> sub = new List<String>(names); // Clone

Set<Id> accountIds = new Set<Id>();
accountIds.add('001XX000003AAA');
accountIds.add('001XX000003AAA'); // Ignored!

// Perfect for: collecting unique IDs
Set<Id> parentIds = new Set<Id>();
for (Contact c : contacts) {

// Use in SOQL: WHERE Id IN :parentIds

// Convert List → Set (remove dupes)

Asynchronous Apex — Beyond the Transaction

When you need higher limits, callouts from triggers, or scheduled processing — use async Apex. Each method has different strengths.

@future — Simple Async

// Fire-and-forget. No return value. No chaining.
public class CalloutService {
    @future(callout=true)
    public static void sendToExternal(Set<Id> recordIds) {
        // Higher limits: 60s timeout, 12MB heap
        List<Account> accts = [SELECT Name, Phone
            FROM Account WHERE Id IN :recordIds];
        // Make HTTP callout...
    }
}

// Limitations:
// - Can't call @future from @future
// - Can't call from Batch
// - Parameters: only primitives or collections of primitives
// - No tracking (fire & forget)

Queueable — Modern Async

public class ProcessRecordsJob implements Queueable {
    private List<Account> accounts;
    
    public ProcessRecordsJob(List<Account> accts) {
        this.accounts = accts;
    }
    
    public void execute(QueueableContext ctx) {
        // Process records...
        // Can chain: enqueue another job
        if (moreWork) {
            System.enqueueJob(new ProcessRecordsJob(remaining));
        }
    }
}

// Enqueue:
Id jobId = System.enqueueJob(new ProcessRecordsJob(accts));

// Advantages over @future:
// ✅ Can pass complex objects (sObjects, custom classes)
// ✅ Can chain jobs (one after another)
// ✅ Returns a Job ID for monitoring
// ✅ Can implement Database.AllowsCallouts

Batch Apex — Large Data Volumes

public class CleanupBatch implements 
    Database.Batchable<SObject>, Database.Stateful {
    
    public Integer recordsProcessed = 0; // Stateful!
    
    // 1. Collect records to process
    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator(
            'SELECT Id FROM Account WHERE LastModifiedDate < LAST_YEAR'
        );
    }
    
    // 2. Process in chunks (default 200)
    public void execute(Database.BatchableContext bc, 
                        List<Account> scope) {
        delete scope;
        recordsProcessed += scope.size();
    }
    
    // 3. Finish (send email, chain next batch)
    public void finish(Database.BatchableContext bc) {
        System.debug('Processed: ' + recordsProcessed);
    }
}

// Execute (200 records per batch by default):
Database.executeBatch(new CleanupBatch(), 200);
// For callouts: use batch size 1-100

Exception Handling & Best Practices

Exception Hierarchy

// Apex exception types
try {
    insert accounts;
} catch (DmlException e) {
    // Specific DML failure
    System.debug('DML Error: ' + e.getDmlMessage(0));
    System.debug('Field: ' + e.getDmlFields(0));
    System.debug('Status: ' + e.getDmlStatusCode(0));
} catch (QueryException e) {
    // SOQL failure (no rows, too many rows)
    System.debug('Query Error: ' + e.getMessage());
} catch (CalloutException e) {
    // HTTP callout failure
    System.debug('Callout Error: ' + e.getMessage());
} catch (Exception e) {
    // Catch-all (always last)
    System.debug('Type: ' + e.getTypeName());
    System.debug('Message: ' + e.getMessage());
    System.debug('Stack: ' + e.getStackTraceString());
} finally {
    // Always runs (cleanup)
}

Custom Exception

public class ServiceException extends Exception {}

// Usage:
if (accounts.isEmpty()) {

}

// AuraHandledException (for LWC)
throw new AuraHandledException('User-friendly message');

Apex Best Practices Checklist

  • Bulkify everything — no SOQL/DML inside loops
  • One trigger per object — use handler classes
  • with sharing by default — security first
  • 75%+ test coverage — with meaningful assertions
  • Use maps for lookups — O(1) vs O(n) loops
  • Avoid hardcoded IDs — use Custom Labels/Custom Metadata
  • Separation of concerns — Trigger → Handler → Service → Selector
  • Null-check everything — prevent NullPointerException
  • Limit-aware patterns — check Limits class in long operations
  • Meaningful test names — testBulkInsert_200Records_NoErrors

Apex Design Patterns — Enterprise Architecture

Real-world Salesforce projects use layered architecture. These patterns make code maintainable, testable, and scalable.

Service Layer Pattern

// Service class: business logic lives HERE
public with sharing class OpportunityService {
    
    public static void closeWonOpportunities(Set<Id> oppIds) {
        // 1. Validate
        if (oppIds == null || oppIds.isEmpty()) return;
        
        // 2. Query (use Selector)
        List<Opportunity> opps = OpportunitySelector.getByIds(oppIds);
        
        // 3. Business logic
        List<Opportunity> toUpdate = new List<Opportunity>();
        for (Opportunity opp : opps) {
            if (opp.Amount > 0 && opp.HasProducts__c) {
                opp.StageName = 'Closed Won';
                opp.CloseDate = Date.today();
                toUpdate.add(opp);
            }
        }
        
        // 4. DML (use Unit of Work pattern for complex)
        if (!toUpdate.isEmpty()) {
            update toUpdate;
        }
    }
}

// Called from: Trigger, LWC Controller, Batch, Flow

Selector Pattern

// Centralize ALL queries for an object
public with sharing class OpportunitySelector {
    
    public static List<Opportunity> getByIds(Set<Id> ids) {
        return [SELECT Id, Name, StageName, Amount,
                       CloseDate, HasProducts__c,
                       Account.Name
                FROM Opportunity
                WHERE Id IN :ids];
    }
    
    public static List<Opportunity> getOpenByAccount(Id accountId) {
        return [SELECT Id, Name, Amount, StageName
                FROM Opportunity
                WHERE AccountId = :accountId
                AND IsClosed = false
                ORDER BY CloseDate ASC];
    }
}

Modern Apex Features (2023-2026)

These features are now GA and widely used in production. Interviewers expect you to know them.

Safe Navigation Operator (?.) 

// BEFORE: Null checks everywhere
String name = null;
if (acc != null && acc.Owner != null) {
    name = acc.Owner.Name;
}

// AFTER: Safe navigation (Spring '21+)
String name = acc?.Owner?.Name;
// Returns null if ANY part is null — no NullPointerException!

// Works with methods too:
Integer len = myString?.length();
// Returns null (not exception) if myString is null

// Works with collections:
String first = myList?.get(0)?.Name;

// Works in SOQL:
List<Contact> cons = [SELECT Account?.Name FROM Contact];

// Common patterns:
String msg = error?.body?.message ?? 'Unknown error';
Boolean isActive = record?.IsActive__c == true;

Null Coalescing Operator (??)

// Returns left side if not null, otherwise right side
String name = inputName ?? 'Default Name';
// Equivalent to: inputName != null ? inputName : 'Default'

// Chain multiple fallbacks:
String label = customLabel ?? fieldLabel ?? 'Untitled';

// With safe navigation:
String city = account?.BillingCity ?? 'No City';

// In assignments:
Integer count = existingCount ?? 0;

Switch Statement

// Clean alternative to long if/else chains
switch on stage {
    when 'Prospecting', 'Qualification' {
        priority = 'Low';
    }
    when 'Proposal', 'Negotiation' {
        priority = 'Medium';
    }
    when 'Closed Won' {
        priority = 'Completed';
    }
    when else {
        priority = 'Unknown';
    }
}

// Switch on SObject type:
switch on record {
    when Account acc {
        System.debug('Account: ' + acc.Name);
    }
    when Contact con {
        System.debug('Contact: ' + con.Email);
    }
    when null {
        System.debug('Null record');
    }
    when else {
        System.debug('Other type');
    }
}

// Switch on Enum:
switch on season {
    when WINTER { return 'Cold'; }
    when SUMMER { return 'Hot'; }
    when else { return 'Moderate'; }
}

User Mode Database Operations

// Enforce FLS + CRUD + Sharing in SOQL (Spring '23+)
// BEFORE: with sharing class + manual checks
List<Account> accts = [
    SELECT Name, Phone, Secret_Field__c
    FROM Account
    WITH USER_MODE  // Strips fields user can't see!
];

// Database class with AccessLevel:
List<Account> results = Database.query(
    'SELECT Name FROM Account',
    AccessLevel.USER_MODE  // Enforces all security
);

// DML with User Mode:
Database.insert(records, AccessLevel.USER_MODE);
Database.update(records, AccessLevel.USER_MODE);

// System Mode (bypass security — use carefully):
Database.query(q, AccessLevel.SYSTEM_MODE);

// stripInaccessible (alternative approach):
SObjectAccessDecision decision = Security.stripInaccessible(
    AccessType.READABLE, records
);
List<Account> sanitized = decision.getRecords();
// Fields user lacks FLS for are removed

Interview tip: Always mention USER_MODE as the modern best practice for enforcing security. It replaces manual FLS/CRUD checks and WITH SECURITY_ENFORCED.
